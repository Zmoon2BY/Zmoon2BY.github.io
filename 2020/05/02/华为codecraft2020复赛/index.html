苟进复赛

赛题要求

赛题其他要求不变，在之前找3-7环的基础上，加入以下条件

转账金额浮动区间约束为[0.2, 3] ：循环转账的前后路径的转账金额浮动，不能小于0.2，不能大于3。如账户A转给账户B 转X元，账户B转给账户C转Y元， 那么进行循环检测时，要满足0.2 <= Y/X <= 3的条件。

同时，边数为200w，符合条件的环为2000w

对边权加以限制，总的来说改变并不大，算法流程方面依旧选择了反向3层做可达剪枝 正向7层直接搜索的思路

数据载入

读入方面，选择了mmap多线程读入的方式，对数据进行分4块处理(对于之前的28w边来说效果不大，对于200w的效果提升尚未进行测定)

由于需要映射，所以all（所有的id）需要进行排序，目前未想到有效解决方法

建图

建图过程加入互斥锁，实现多线程建图，多线程对图的邻接表排序等，对于star（起点剪枝）需要进行排序以保证答案的有序性，目前未想到有效解决方法

建图 有三个不同的图，分别为

G 原图 存图方式为vector<结构体>[ 点数 （目前用最大边数200w代替）] 结构体内容为 边的终点与边权

Ginv 原图的反向图 存图方式一样 依旧保存边的终点与边权

Gt 与之前的Gt一样 此处只保存 边的终点忽略边权 用于可达性剪枝

目前方案：排序方式依旧是按照点id从大到小，原因是保证答案的有序，避免对答案的排序（是否最优存疑？）

另一种思路：对边权排序，从而根据[0.2,3]的条件限制二分到大于等于0.2 * val 的位置，但是id顺序是乱的，排序较为耗时（不打算启用）

其他思路：暂无

找环算法 + 输出

找环算法与之前的思路类似，依旧是每次搜索以root为第一位的满足条件的环

先Gt中三层反向搜索，标记可达 reach

正向搜索过程中，首先遍历第一层时，可以根据第一条出边的边权确定起点root的入边边权范围，如果没有满足条件的入边则枚举下一条出边，即假设有环 …… y - > root - > x …… - >y 则选择root -> x作为出边时，val [root,x] / val [y,root]满足[0.2,3]约束的边y - > root 是可以确定的，这样可以减少一部分搜索，属于“后向判断”

“前向判断”发生在正向搜索过程之中，比如有root - > u1 - > u2 - > u3 … 则应当依次正向判断相邻两边是否满足条件约束，后向判断可以看作是前向的剪枝？比如root - > u1 - > u2 - > root 如果只采用前向判断，假设 root - > u1 与u1- > u2 是满足的，u1- > u2 与 u2- > root 是满足的，虽然此时搜索已经发现环了，但是对于整个环究竟满足条件与否还未能确定，因为 u2- > root 与root - > u1是否满足条件我们还不知道，后向判断可以解决此类问题

后向判断需要Ginv，前向判断在G中，同时在第6层可以通过后向判断的标记以及存边权的方式可以减少搜索，因为在第6层，我们只需要u6 - > root这一条边的边权 如果它满足前向与后向的判断那么这个7环就是可行的

以上算法流程与之前初赛的算法流程执行时间大致相仿

整个算法的目的就是为了用上[0.2,3]这一条件，同时不要让复杂度爆炸，难点在于只用前向判断， …… y - > root - > x …… - >y 边y - >root 与 root - x 是否满足约束条件

然后找环算法过程中并未采取直接存char的方式，虽然这种方式初赛线下爆慢，线上居然有所提升，我认为这是数据集造的不好的问题，依旧是先存int，找完环之后把结果转为char

整个找环过程多进程对id分段进行执行算法，进程间通信用的是共享内存mmap，以及进程间同步采用信号量的方式，主进程等待子进程 找环 + 答案int转char并分段拼接 + 把找环结果与字符串长度写入共享内存 这三步。主进程启动开始进行对各个子进程汇报信息的汇总并且做输出的准备工作，子进程挂起等待，主进程汇总答案结果，主进程计算各子进程应当开始载入数据的位置，mmap映射输出文件，调整文件大小，并将载入数据的位置信息写入共享内存，主进程任务结束等待子进程完成输出。子进程读共享内存的载入数据的位置信息，对对应的位置进行写入数据

输出方面，mmap用的并不熟练，不知道是不是最优做法

总结

多线程输入 多线程建图 多进程找环 多进程输出
